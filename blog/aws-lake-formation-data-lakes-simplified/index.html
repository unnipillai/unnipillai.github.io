AWS Lake Formation = Data Lakes Simplified

From months to days — what AWS Lake Formation actually changes

UP
Unni Pillai December 4, 2018

Over the last couple of years, I have met numerous customers who are building data lakes to centrally store, process & enable self-service analytics and machine at scale.

A data lake project usually requires them to set up a unified data ingestion layer, figuring out how and where data will be stored, cataloging the datasets & securing them to comply with security needs.

Using AWS managed services like Kinesis, Database Migration Service, S3, Glue, EMR, Athena & Redshift has helped them attain their vision of building a data lake. A medium sized project with a couple of data sources & user personas will take few months to build this out with the right level of security & governance in place.

In re:invent 2018, AWS launched AWS Lake Formation, this service makes it easy for AWS customers to build secure, scalable & flexible data lakes in days instead of months. You can choose from a collection of blueprints, define transformations, storage, and security permissions to spin up the baseline needed to build your enterprise data lake.

Lake Formation protects your data by giving you a central location where you can configure granular data access policies that protect your data, regardless of which services are used to access it. Lake Formation policies apply across all the AWS services that are accessing data from your lake. You can configure users, groups, roles and define the policies these roles can access, down to the table and column level.

With AWS Lake Formation in place, organizations can focus on what matters most (building analytics capability) and avoid undifferentiated (infra, transformations, etc).

Very excited to see how AWS customers will be using this services to build their data lake foundation to accelerate their data driven decisions.
